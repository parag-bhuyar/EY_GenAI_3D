LLM Mechanics for Practitioners
Before you build production prompts for Meridian Retail Bank, you need to know what actually happens between a request and a response — and how to control it on purpose instead of by accident.

The one idea underneath everything else in this module: an LLM is a next-token predictor. It doesn't "know" your answer — at every step it scores every possible next token and picks one. Everything else in this module is a consequence of that single fact: tokens are the unit it thinks in, decoding controls decide how it picks from its own scored candidates, and determinism is just "how much randomness did we allow in that pick." Unit 2's playground below lets you run that exact pick-a-token step yourself.

Anatomy of a response — what your instructor will inspect live

Every Chat Completions call returns one object with three things worth looking at together: the text, how it ended, and what it cost you in tokens.

response.choices[0].message.content # -> "I'm sorry to hear your card was declined..." response.choices[0].finish_reason # -> "stop" (finished naturally) or "length" (hit max_tokens) response.usage.prompt_tokens, response.usage.completion_tokens # -> 42, 31 — this pair is what you're billed on, every single call

"Cost" is tokens × the provider's current per-token rate — deliberately not shown as a number here (see Unit 4's note on why). The shape above is what you'll actually inspect in today's lab and every lab after it.

Context window = working memory

The context window is the maximum number of tokens (prompt + response combined) a model can hold in one call. Feed it a 200-page policy document and a short question, and you can blow the budget before the model even answers — this is exactly the problem Day 2's chunking and retrieval solve.

"It fits" isn't the end of the story

Prompt caching — several providers now discount or speed up a repeated prefix (a stable system prompt, for instance) instead of re-billing it in full every call. Lost-in-the-middle — a model doesn't attend equally to everything in a large context, so fitting a document isn't the same as the model actually using all of it well (Day 2 covers this properly).

Why this matters at Meridian Retail Bank

ScenarioToken cost
Short customer FAQ replyCheap — tens to low hundreds of tokens
Pasting a full loan policy document into every promptExpensive and wasteful — most of it is irrelevant to any one question
A long multi-turn support conversationGrows every turn — history has to be managed, not just appended forever
🧠 A colleague wants to paste Meridian's entire 40-page product terms PDF into the system prompt for every customer query. What's the problem?
Two problems at once: cost (every single call now pays for ~40 pages of tokens whether the question needs them or not) and context budget (a long enough document plus a long conversation history can exceed the model's context window entirely). This is exactly why Day 2 exists — retrieve only the relevant passage instead of stuffing the whole document in every time.

Unit 3
Message Roles & the Request-Response Loop
system / user / assistant — and why every enterprise integration keeps them strictly separated.

The three roles

system — sets standing behaviour for the whole conversation ("You are a Meridian Retail Bank support assistant; never promise a refund"). user — the actual request this turn. assistant — the model's own prior replies, fed back in on multi-turn calls so it has conversational memory.

The request-response loop

Every call is stateless on the server side — the model has no memory between calls. "Conversation" is an illusion your code maintains by resending the growing message list every time. This is worth internalising now: M1.2's structured-extraction function and every later agent both live and die by getting this loop right.

🧠 A 20-turn support conversation keeps resending the full history every call. What's this doing to your token bill, and what would you do about it?
Cost grows roughly linearly with turn count, since each call re-sends everything before it — turn 20 pays for turns 1-19 all over again. Production systems manage this with a sliding window (drop the oldest turns), periodic summarisation (compress old turns into a short summary message), or both. This is the same "don't just append forever" problem Unit 1's context-window discussion raised — roles and context budget are the same constraint wearing two hats.

Unit 5
Determinism, Variance & Reproducibility
The same prompt can produce a different answer twice — by design, not by bug.

Why this is a feature, not a bug

Sampling is what makes a model's writing feel natural instead of robotic — but for a regulated banking FAQ responder, unpredictable phrasing is a liability, not a feature. If you skipped Unit 2's Sampling Playground, go back and click "Sample next token" five times at temperature 0, then five times at 1.5 — you'll see this directly instead of taking it on faith.

Where it bites in practice

Two identical support tickets get subtly different replies. A regression test that "randomly" fails. An audit log where the same input produced a different recommendation on two different days. All three are the direct, predictable consequence of the decoding controls in Unit 2 — this module's lab quantifies exactly how much.

🧠 For Meridian's loan-eligibility triage agent (Day 3), would you want high or low temperature — and why?
Low — ideally temperature 0, optionally with a fixed seed. A triage decision needs to be consistent and auditable: the same applicant profile submitted twice should get the same reasoning and the same recommendation, not a coin flip in the model's sampling. High temperature is the right call for tasks where variety is a feature (marketing copy, brainstorming), not for anything a regulator might ask you to reproduce.
