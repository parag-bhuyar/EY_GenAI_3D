Unit 1
SDK Setup, Auth & Environment Hygiene
The unglamorous part that determines whether anything else in this programme actually runs.

Never hard-code a key

Your API key goes in an environment variable, read at runtime — never typed into a notebook cell, never committed to git. Every lab in this programme, starting with M1.1, assumes OPENAI_API_KEY is already set in your shell before you open anything.

Fail loudly if the key's missing

Today's starter code asserts the key exists before doing anything else. That's deliberate — a script that silently proceeds and fails 30 seconds later on the actual API call wastes far more of your time than one that stops immediately with a clear message.

✗ client = OpenAI(api_key="sk-abc123...") ← never do this ✓ client = OpenAI() ← reads OPENAI_API_KEY from the environment automatically
Unit 2
Streaming Responses
Why a chat UI doesn't wait for the whole answer before showing anything.

Non-streaming

The call blocks until the entire response is generated, then returns it all at once. Simpler to write, but for a long response the user stares at a blank screen the whole time.

response = client.chat.completions.create( model="gpt-4o-mini", messages=[...], ) print(response.choices[0].message.content)

Streaming

The same call, but tokens arrive incrementally as they're generated — the classic "typing" effect. Requires handling partial chunks instead of one clean object.

stream = client.chat.completions.create( model="gpt-4o-mini", messages=[...], stream=True, ) for chunk in stream: delta = chunk.choices[0].delta.content if delta: print(delta, end="")
🧠 Today's structured-extraction lab uses tool calling, not streaming. Why doesn't streaming make sense there?
Structured extraction needs one complete, valid JSON object before it's useful — a partially-streamed JSON fragment can't be validated or parsed mid-stream. Streaming is for improving perceived latency on free-text responses a human is reading live; it doesn't help when the consumer is your own validation code waiting for a complete structured object.

Unit 4
Function / Tool Calling
Defining a tool schema and routing the model's tool call — today's lab, end to end.

The three-step loop

  1. Send the message with tools=[...] and (usually) tool_choice forcing a specific tool
  2. The model returns a tool call with a JSON arguments string instead of plain text
  3. Your code parses and validates those arguments — never trust them blindly, the model can still return a malformed or out-of-schema value
Unit 5
Robustness: Retries, Timeouts & Rate Limits
The difference between a lab exercise and something you'd trust in production.

What can go wrong on a real call

A network blip. A rate limit during a busy period. The model returning a tool call with a value outside your schema's allowed options — this last one is a validation failure, not a network failure, and today's lab specifically builds a retry loop for exactly this case.

Retry ≠ retry forever

A bounded retry count with backoff, then a clear failure — never a silent infinite loop, and never a failure that just returns garbage instead of raising. Today's lab's max_retries parameter and final RuntimeError are the minimum viable version of this pattern.
