Unit 1
What Embeddings Are: Meaning Becomes Geometry
Every piece of text becomes a list of numbers — positioned so that similar meaning ends up close together.

The core idea

An embedding model converts text into a fixed-length vector (1536 numbers for text-embedding-3-small) — a point in a very high-dimensional space. Text with similar meaning ends up at nearby points; unrelated text ends up far apart. Nobody designed these dimensions by hand — the model learned them from data.

Why this is useful

Once "meaning" is a position in space, finding "the most relevant passage for this question" becomes a geometry problem: which stored point is closest to the query's point? That's the entire idea behind semantic search — everything else in this module is detail on top of that one idea.

Unit 2
Similarity Metrics: Cosine Similarity
The specific way "closeness" gets measured in today's lab.

Reading a cosine similarity score

RangeRough meaning
~1.0Nearly identical meaning (or the exact same text)
~0.5–0.8Related, often what a good semantic-search match looks like in practice
~0.0–0.3Mostly unrelated text — typical for two random sentences
NegativeRare in practice with real embeddings; would indicate near-opposite meaning

There's no universal "similar enough" cutoff — a score is only meaningful relative to the other candidates in the same search, which is exactly why semantic_search ranks and takes a top-k rather than checking against a fixed threshold.

Unit 3
Generating Embeddings with text-embedding-3
This programme's confirmed default embedding model — cheap enough to embed generously.

One call, one vector

response = client.embeddings.create( model="text-embedding-3-small", input="some banking passage text", ) vector = response.data[0].embedding # vector is a list of 1536 floats

No output tokens, ever

Unlike a chat completion, an embedding call has no "response text" to generate — you only pay for input tokens. This is a big part of why embeddings are so cheap relative to chat completions, and why today's lab can afford to embed 10 passages + 5 queries + a handful of demo sentences without a second thought.

A real cost example — embedding the whole lab

At text-embedding-3-small pricing — $0.02 / 1M input tokens, verified 2026-07-19, re-check before quoting a live cohort — embedding all 10 Meridian passages plus all 5 fixed queries (roughly 550 tokens total) costs approximately $0.000011 — a thousandth of a cent. Even running the autograder repeatedly while debugging costs, for practical purposes, nothing; the real budget concern in this programme is chat-completion calls (M1.1–M1.3), not embeddings.

Unit 4
Chunking Intuition & Its Effect on Retrieval Quality
An embedding represents the meaning of a whole block of text as ONE vector — mixing topics dilutes it.

One passage, two topics — unchunked

A passage covering both savings rates AND loan eligibility gets embedded as a single vector that has to represent both topics at once. A query specifically about loan eligibility now competes against the "savings rate" content diluting that same vector.

The same content, chunked by topic

Split into a savings-only chunk and a loan-only chunk, each vector represents one focused idea. Today's lab's stretch task measures this directly: the loan-specific chunk should score higher against a loan-specific query than the combined, unchunked version does.

🧠 If chunking by topic generally helps, why not just chunk every document down to single sentences?
Chunks that are too small lose context — a single sentence torn out of a policy document may be ambiguous or misleading on its own (e.g. "this fee is waived" without the eligibility condition that follows it). Chunking is a trade-off between precision (smaller, more focused chunks) and context (enough surrounding text for the chunk to still make sense standalone) — not "smaller is always better." Day 2's M2.3 RAG pipeline is where this trade-off gets tuned properly.

Unit 5
Semantic vs. Keyword Search: Where Each Wins
Neither is universally better — today's lab runs both against the same queries so you can see exactly where they diverge.
ScenarioKeyword searchSemantic search
Query uses the exact same words as the sourceWorks well, fast, cheapAlso works, but is overkill
Query paraphrases the source ("machine" vs "ATM")Often fails — no literal overlapWorks — meaning matches even without shared words
Exact ID / code / account-number lookupWins — exact match is exactly what's neededCan be unreliable — embeddings aren't built for exact-match precision
Very large document set, need for filtering by metadataFast at scale, but shallowNeeds an indexed vector store (Day 2's M2.2) to stay fast at scale

Production RAG systems often combine both (hybrid search) — today's lab isolates them so the trade-off is visible, not because production should pick only one.
