## How to work through this file

1. Open `Day1/labs/starter/D1_M1.2_OpenAI_API_Starter.py`.
2. Read the `CustomerRequest` schema first — this is the contract every extraction must satisfy.
   Notice this is Pydantic doing two jobs at once: it's a data container, *and* calling
   `CustomerRequest(**some_dict)` actively validates every field.
3. Read `EXTRACTION_TOOL` — this is the JSON-schema description of `CustomerRequest`'s fields that
   actually gets sent to the model. Notice every field is `required`, and the `enum` lists mirror
   the schema's `Literal` options exactly.
4. Read `extract_customer_request` **line by line** — every step has a comment explaining what
   it's doing and why: forcing the tool call, parsing the JSON arguments string, validating with
   Pydantic, and the retry loop that catches a validation failure instead of crashing.
5. Run the file: `python Day1/labs/starter/D1_M1.2_OpenAI_API_Starter.py`. It should print a
   validated JSON record for the sample message at the bottom of the file.

## What to actually pay attention to

- **Why `tool_choice` forces the tool** instead of leaving it to `"auto"` — an extraction task
  should never "decide" not to extract. (Day 3's agents are where letting the model choose between
  tools is the right pattern — not here.)
- **Why the arguments are parsed *and* validated**, not just parsed. `json.loads()` only tells you
  it's valid JSON; `CustomerRequest(**parsed)` is what actually checks the fields are the right
  type and, for constrained fields like `account_type`, one of the allowed values.
- **What the retry loop actually catches**: a `ValidationError` (model returned a value outside an
  allowed `Literal`), a `ValueError` (no tool call came back at all), or a `json.JSONDecodeError`
  (malformed JSON string) — and what it does *not* do, which is retry forever. After `max_retries`
  failures it raises a clear `RuntimeError` instead of looping silently.

## Running the autograder

```
python Day1/autograder/D1_M1.2_OpenAI_API_Autograder.py
```

This first checks your error-handling by simulating one malformed model response (no real API
call for that part) and confirming `extract_customer_request` recovers instead of crashing. It
then runs your extraction against 10 real sample customer messages
(`Day1/data/D1_M1.2_sample_customer_messages.json`) — this makes 10 real API calls, so it costs a
small amount each run; don't run it in a loop while debugging, use the manual smoke test (above)
first. You need at least 8/10 to pass. On pass, push the result:

```
git add Day1/submissions/D1_M1.2.json
git commit -m "D1 M1.2 submission"
git push origin main
```

Extracting all 10/10 correctly counts as this module's stretch/Diamond signal — no separate
stretch task to build.
