## The situation

In M1.2 you built a function that turns a messy customer message into clean, structured data. Today's lab is different: Meridian Retail Bank's FAQ bot went live with a plain, one-sentence prompt — and overnight it got talked into ignoring its own rules, answered off-topic questions, and nearly handed over another customer's account balance. This file shows exactly how that got fixed, one guardrail piece at a time.

## How to work through the notebook

Open `Day1/labs/starter/D1_M1.3_Prompt_Engineering_Starter.ipynb` and run the setup cell first.

1. **Part 1 — the incident log, then role framing.** The naive prompt's real response to 4 incidents (an off-topic question, two injection attempts, a privacy request), plus why each one happened. Then Round 1 narrows the bot's job to Meridian FAQs only, re-tested against the weather question. *You should see `status=invalid_shape` — that is expected, and Part 2 is why.*
2. **Part 2 — output contract.** Adds a strict JSON shape (`status`/`topic`/`response`). Re-tested against the same case. *Now you get a real status value.*
3. **Part 3 — refusal rule.** Explicitly lists what to decline. Re-tested against 3 incidents at once. *All three should refuse. Case 8 will still be open.*
4. **Part 4 — injection hardening (final).** Stops a crafted message from talking the bot out of its own rules. Re-tested against both injection attempts, plus two tougher stretch variants. *This is the round that closes case 8.*
5. **Part 5 — versioning and evaluation.** All five versions side by side, why each of the 5 test-case categories exists, and the graded pass/fail report against the final hardened prompt.
6. **Run the last cell** to save `m1_3_eval_results.json`, then run the autograder:
   `python Day1/autograder/D1_M1.3_Prompt_Engineering_Autograder.py`

## What to actually pay attention to

- Which round fixes which incident — Round 3 closes 3 of the 4 incidents; Round 4 is what specifically closes the two injection attempts, because a refusal rule alone can still be argued with.
- The output contract (format) and the refusal rule (safety) are two different guarantees — a prompt can nail one and still fail the other.

## Running the autograder

```
python Day1/autograder/D1_M1.3_Prompt_Engineering_Autograder.py
```

This makes 10 real API calls against the hardened prompt (a small cost each run) and checks three things: (1) the prompt is genuinely versioned, (2) the two prompt-injection cases and the privacy-breach case are **all** correctly refused — this is checked separately and is a hard requirement, not averaged into the overall score, and (3) at least 8 of the 10 cases overall match their expected outcome.
