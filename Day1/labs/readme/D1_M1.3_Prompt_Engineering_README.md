# D1 · M1.3 — Prompt Engineering Patterns for Production

**Duration:** 1.5 hrs · **Type:** Individual, run-and-understand, autograded + live peer review · **Capstone:** this lab's guard-railed responder is Capstone Phase 1 material (prompt-engineered query handling)

## Objective

This starter file already contains a **versioned prompt template** (a naive `v1` and a hardened
`v2`) and a fully working **10-case prompt evaluation set** for a Meridian Retail Bank FAQ
responder — the two deliverables this module actually produces. You are not writing prompts from
scratch. You are watching the naive version fail on tricky inputs, watching the hardened version
hold on the exact same inputs, and running the full evaluation harness that grades it.

## Prerequisites

- M1.1 and M1.2 completed.
- `OPENAI_API_KEY` set as an environment variable.
- `student_config.json` created at the repo root with your real name and GitHub username.

## How to work through this file

1. Open `Day1/labs/starter/D1_M1.3_Prompt_Engineering_Starter.py`.
2. Read `PROMPT_VERSIONS["v1_naive"]` and `PROMPT_VERSIONS["v2_hardened"]` side by side, along with
   `VERSION_LOG` above them — this is what "versioned prompt template" actually means: not a
   single string, but a record of what changed and why.
3. Read `ask_faq_responder` — notice `json_mode` isn't always on: Task 1 deliberately calls the
   naive prompt in plain text (no output contract at all) so you can see how unstructured and
   unpredictable an unhardened response really is.
4. Run the file: `python Day1/labs/starter/D1_M1.3_Prompt_Engineering_Starter.py`.
5. **Task 1** prints the naive prompt's raw output on 4 tricky inputs (an out-of-scope question,
   two prompt-injection attempts, a privacy-breach request). Read these before moving on — this is
   the "before" picture.
6. **Task 2** prints the hardened prompt's validated output on the *same* 4 inputs — the "after"
   picture, same inputs, different prompt.
7. **Task 3** runs the full 10-case evaluation harness and prints a pass/fail report — this report
   is the module's actual Expected Output per the programme syllabus.
8. **Task 4** (in the `__main__` block) — replace the reflection placeholder with your own answers
   to the three guided questions, based on what you actually saw in Tasks 1–3.
9. **Task 5 (stretch, optional, Diamond tier)** — two tougher injection variants, not in the base
   evaluation set. Only attempt this if you've finished Tasks 1–4 with time to spare.

## What to actually pay attention to

- **Why the naive prompt's Task 1 output looks the way it does** — no output contract means no
  guaranteed structure; no refusal rule means it may partially engage with an injection attempt
  instead of declining.
- **Which specific addition in `v2_hardened`** (role framing / output contract / refusal rule /
  injection hardening) you think mattered most for each tricky case — this is exactly what the
  live peer review afterward asks you to defend.
- **The difference between an output contract and a guardrail** — the output contract (`status`/
  `topic`/`response` JSON shape) is about *format*; the refusal rule and injection hardening are
  about *content/safety*. A prompt can have a perfect output contract and still leak on guardrails,
  or vice versa — Task 2/3 test both.

## Running the autograder

```
python Day1/autograder/D1_M1.3_Prompt_Engineering_Autograder.py
```

This makes 10 real API calls against the hardened prompt (a small cost each run) and checks three
things: (1) the prompt is genuinely versioned, (2) the two prompt-injection cases and the
privacy-breach case are **all** correctly refused — this is checked separately and is a hard
requirement, not averaged into the overall score, and (3) at least 8 of the 10 cases overall match
their expected outcome. On pass, push the result:

```
git add Day1/submissions/D1_M1.3.json
git commit -m "D1 M1.3 submission"
git push origin main
```

A perfect 10/10 counts as this module's stretch/Diamond signal.

## Live peer review

After the lab, you'll pair up with another participant for a **fast 2–3 minute swap**: show each
other your Task 1 vs Task 2 output for one tricky case, and each give one specific piece of
feedback on the other's guardrail reasoning. Full facilitation format is run by the trainer — just
have your Task 1/2/3 output ready to show.
