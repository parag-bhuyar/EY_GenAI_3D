## Objective

## How to work through this file

1. Open `Day1/labs/starter/D1_M1.4_Embeddings_Starter.py`.
2. Read `get_embedding` and `cosine_similarity` first — these two functions are the entire
   mechanism. `cosine_similarity` is implemented by hand (no numpy) so you can see the actual
   dot-product-over-magnitudes formula, not just call a library function.
3. Run the file: `python Day1/labs/starter/D1_M1.4_Embeddings_Starter.py`.
4. **Task 1** embeds 3 simple sentences and ranks them against 1 query — the smallest possible
   version of "meaning becomes geometry." Confirm the banking sentence ranks highest before moving on.
5. **Task 2** embeds all 10 real Meridian Retail Bank passages into an in-memory index (10 API
   calls) — read the comment explaining why this is fine at 10 passages and why it wouldn't scale.
6. **Task 3** is `semantic_search` itself — read it carefully, it's the module's actual Deliverable.
7. **Task 4** runs the same 5 fixed queries through both `semantic_search` and a deliberately naive
   `keyword_search`, side by side. Pay attention to query #4 specifically.
8. **Task 5** runs the full graded evaluation — 5 fixed queries, checking whether the #1 result
   matches the expected passage.
9. **Task 6 (stretch, optional, Diamond tier)** — a chunking experiment showing why combining two
   unrelated topics into one embedded passage hurts retrieval precision.

## What to actually pay attention to

- **Why brute-force comparison is fine here but wouldn't scale** — `semantic_search` compares the
  query against every single passage embedding, one at a time. At 10 passages this is instant; at
  10 million it would not be. This is exactly the gap Day 2's M2.2 (Chroma/Qdrant, HNSW indexing)
  fills — same underlying idea (embed once, compare many times), different storage/search layer.
- **Where semantic search actually wins over keyword search** in Task 4 — specifically look at
  which query has the least literal word overlap with its correct passage, and notice that's often
  where keyword search struggles most.
- **What the chunking experiment (if you run it) shows** — an embedding represents "the meaning of
  this whole block of text" as one vector; mixing two unrelated topics into one block dilutes both.

## Running the autograder

```
python Day1/autograder/D1_M1.4_Embeddings_Autograder.py
```

This first checks that `cosine_similarity` is mathematically correct (no API calls needed for
that part), then rebuilds a fresh passage index and runs the 5 fixed queries against it — 15 real
embedding calls total, very cheap (embeddings pricing is far below chat-completion pricing) but
not free. You need at least 4/5 fixed queries to retrieve their expected passage at #1. On pass,
push the result:

```
git add Day1/submissions/D1_M1.4.json
git commit -m "D1 M1.4 submission"
git push origin main
```

A perfect 5/5 counts as this module's stretch/Diamond signal.
