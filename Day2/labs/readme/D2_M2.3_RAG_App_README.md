## Where this fits

The M2.3/M2.4 notebook taught you retrieval on a corpus that was already prepared for you: 40 short articles, one article per chunk, ready to embed.

Real documents do not arrive like that. They arrive as a 60-page HR policy PDF.

This lab covers the part the notebook skipped — **getting real documents into a vector store in the first place**:

| Notebook (M2.3/M2.4) | This lab |

Together they are the full picture: this lab is the *ingestion* half, the notebook is the *retrieval* half.

## What you'll build

A Streamlit web app that answers questions about PDFs you upload. Upload a document, build a searchable index, ask questions in plain English, and get answers grounded in the actual text — with the source chunks shown underneath.

Two PDFs are included to start with: a corporate HR policy document and the NASSCOM HR manual.

## The four things worth understanding in the code

Everything lives in two small files. Read them in this order.

### 1. Loading — `rag/indexer.py`, `chunk_document()`

```python
loader = PyPDFLoader(str(path))
documents = loader.load()
```

A **loader** turns a file format into LangChain `Document` objects. `PyPDFLoader` gives you one Document per page, with the page number kept in metadata. There are loaders for Word, HTML, CSV, Notion, Confluence and dozens more — they all produce the same `Document` shape, which is the point.

### 2. Chunking — the two numbers that matter most

```python
RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)
```

A whole page is too big to embed usefully — one vector cannot represent four unrelated topics. So documents get split.

- **`chunk_size=1000`** — roughly a long paragraph.
- **`chunk_overlap=200`** — each chunk repeats the last 200 characters of the previous one, so a sentence split across a boundary still appears whole somewhere.

**"Recursive" means it tries to split on natural boundaries first** — paragraphs, then lines, then sentences, then words — and only cuts mid-word as a last resort.

> **Try this:** set `chunk_size=200`, rebuild the index, and ask the same question. Then try `chunk_size=4000`. Both make the answers worse, for opposite reasons. Small chunks lose the context around the answer; large chunks bury it in noise. This one setting causes more bad RAG results than model choice does.

### 3. Persisting — why the index survives a restart

```python
Chroma.from_documents(all_chunks, embeddings, persist_directory="chroma_db")
```

The vectors are written to a `chroma_db/` folder on disk. Stop the app, start it again, and your index is still there — embedding is the slow, expensive step, so you do it once.

Note `index_exists()` checks the collection actually **contains vectors**, not just that the folder exists. An empty folder from a failed run would otherwise look like a working index.

### 4. Retrieval and answering — `rag/retriever.py`

```python
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})
RetrievalQA.from_chain_type(llm=llm, chain_type="stuff",
                            retriever=retriever, return_source_documents=True)
```

- **`k=4`** — fetch the 4 nearest chunks. Compare with the notebook, where you retrieved 20 and re-ranked down to 5. This app has no re-ranker; those 4 go straight to the model.
- **`chain_type="stuff"`** — "stuff" all retrieved chunks into one prompt. The simplest strategy, and the right one when your chunks are small.
- **`return_source_documents=True`** — this is what makes the citations in the UI possible. Without it you get an answer with no way to check it.

## Connect it back to the notebook

Once it's working, ask yourself:

- This app retrieves `k=4` with pure semantic search. Which questions about an HR policy would **keyword** search handle better? (Employee ID numbers, policy codes, section references.)
- There is no re-ranker. Given what you saw in M2.4, where would the correct chunk end up on a hard question?
- There is no metadata filter. If this held policies for three different countries, what would go wrong?

**Those three gaps are exactly what M2.3/M2.4's notebook fixed.** This app is the honest starting point most teams ship first — and then have to improve.

## If something breaks

| Symptom | Cause |
|---|---|
| `ConnectionError` / "connection refused" on indexing | Ollama isn't running. Start it, then check `ollama list`. |
| Indexing works, then answering fails | You pulled `nomic-embed-text` but not `llama3.2:1b`. Pull it. This is the most common failure in this lab. |
| `ModuleNotFoundError: streamlit` | The venv isn't activated, or `pip install -r requirements.txt` was skipped. |
| Answers are vague or wrong | Usually chunking, not the model. Check your `chunk_size`, and read the source chunks the UI shows — if the right text isn't in them, it's a retrieval problem, not a generation problem. |
| `SQLITE_READONLY_DBMOVED` | Something deleted `chroma_db/` while the app was running. Use the app's clear-index button instead, then restart. |
| Answers are slow | `llama3.2:1b` runs on your CPU. 10–30 seconds is normal. It is a 1-billion-parameter model — small and fast by LLM standards, still real work for a laptop. |

## What good looks like

You can explain, without notes, what happens between "I uploaded a PDF" and "I got an answer" — all six steps: load → chunk → embed → store → retrieve → generate. If you can do that, you understand RAG.
