# D2 · M2.3 + M2.4 — Hybrid Search, Retrieval & Re-ranking

## The scenario

Meridian Bank's support assistant went live last month. Priya, head of support, files one bug:

> *"Customers ask about missing salary. The bot answers with account benefits. Fix it before Monday."*

You will find the cause, fix it, and measure the fix.

## Setup

The first cell installs `rank_bm25` and `chromadb`. "Requirement already satisfied" is a good outcome.

Your `OPENAI_API_KEY` must be in a `.env` file. The notebook calls `load_dotenv()` for you — without that line `os.environ` will not see the key even though the file exists on disk.

If the data files are not found, open the notebook from inside the student repo. The notebook walks up the folder tree looking for `Day2/data`.

## What each part does

**Part 1 — Two engines over the same articles**
Builds a BM25 keyword index and a Chroma vector index over the same 40 articles. Runs *"My salary hasn't arrived"* through both.

*You should see:* keyword puts **"Salary account benefits"** at rank 1 — a page that answers nothing but says "salary" six times. That is Priya's bug, reproduced in one cell. Semantic does better but a generic "payment not received" page is right behind the correct answer.

**Part 2 — Fusion**
Normalises both score sets, then merges the two lists two ways: weighted score fusion (with an alpha sweep) and Reciprocal Rank Fusion.

*You should see:* the correct NEFT salary article climbs from rank 4 to rank 1 or 2 under both methods. Watch the RRF table — a document both engines placed reasonably beats a document one engine loved and the other ignored.

**Part 3 — Retrieve wide, re-rank narrow**
Applies a metadata filter (segment + region), retrieves 20 hybrid candidates, then re-ranks down to 5.

*You should see:* the filter removes **"Direct deposit not received — US payroll"**, which is semantically almost perfect and completely wrong for an Indian retail customer. Then the re-ranker promotes an article from deep in the candidate list to rank 1.

**Part 4 — End to end, with a metric**
Answers all five phrasings of the missing-salary question, then measures Recall@1 and Recall@5 for semantic-only versus hybrid + re-ranking.

*You should see:* a real improvement on both metrics — on the trainer's run, Recall@1 went from **70% to 100%** across the 10 evaluation queries. Your numbers may differ by a query or two; the direction is what matters. The notebook prints the status email for you.

## A note on the re-ranker

This lab uses `gpt-4o-mini` as the cross-encoder — one call scores all 20 candidates. It takes 1–3 seconds.

A production re-ranker (Cohere Rerank, or a local BGE / MiniLM cross-encoder) does the same job in around 100 ms. We use the LLM because you already have the key and there is nothing to install. The *ranking behaviour* is what this lab is teaching; production teams pick a real cross-encoder for the latency.

## Cost

Roughly 60 chat calls and 60 embedding calls if you run every cell once, on `gpt-4o-mini` and `text-embedding-3-small`. A few cents.

## If something breaks

| Symptom | Cause |
|---|---|
| `OPENAI_API_KEY is not set` | `.env` is missing, or in the wrong folder. It must be next to the notebook or two levels up. |
| `FileNotFoundError: Could not locate Day2/data` | Notebook opened from outside the student repo. |
| `ModuleNotFoundError: rank_bm25` | Setup cell was skipped, or the kernel is a different Python than your terminal. Check the kernel picker top-right. |
| Chroma `collection already exists` | Re-run the cell — it deletes and recreates the collection each time. |
| Ranking differs slightly from the README | Expected. Embedding runs vary. What matters is the movement, not the exact scores. |
