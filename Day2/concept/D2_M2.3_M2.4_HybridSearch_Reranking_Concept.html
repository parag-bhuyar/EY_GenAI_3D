Concept 2 · Review
Keyword search: matching the words
The oldest trick in search. Still running inside Google, Amazon and every bank you know.

Keyword search counts words. If your query word appears in a document, that document scores points.

Rare words score more than common words. "NEFT" is rare, so it counts a lot. "the" is everywhere, so it counts nothing.

The standard algorithm is called BM25. Think of it as smart word counting.

🎮 Simulation 2 · Word matcher
Watch BM25 count words
Pick a query. Matching words light up. Score = how many rare query words the document contains.

✅ Keyword search is great at

  • Exact IDs: TXN-99213, IFSC HDFC0001234
  • Product names: "Meridian Platinum Card"
  • Rare technical terms: NEFT, RTGS, KYC
  • Speed — it is extremely fast and cheap

❌ Keyword search is blind to

  • Synonyms: paycheck ≠ salary
  • Meaning: "money not credited" ≠ "credit not received"
  • Typos and different languages
  • Anything the writer phrased differently
🤯 Still relevant: Google, Amazon, Elasticsearch and every enterprise search box you have ever used still run BM25 today. Nobody threw it away when embeddings arrived.
Concept 3 · Review
Semantic search: matching the meaning
You built this yesterday. Text becomes numbers. Close numbers mean close meaning.

An embedding model turns any sentence into a list of numbers — a vector.

Sentences with similar meaning get similar vectors. So "paycheck missing" lands near "salary not credited".

The vector store finds the nearest vectors. No shared word needed.

🧠 Quick check: why did "paycheck missing" land near the salary article, with no shared words?
Because the embedding model was trained on huge amounts of text where "paycheck" and "salary" appear in the same situations. It learned they mean nearly the same thing, and gives them nearly the same vector. Distance in vector space is a measure of meaning, not spelling.

🔤 Keyword fails when…

  • The customer uses different words
  • A long, chatty page repeats a common word
  • The question is conceptual, not literal

🧠 Semantic fails when…

  • An exact code or ID matters (TXN-99213)
  • Two products sound similar but are different
  • A rare term carries all the meaning (NEFT vs UPI)
🧠 Quick check: a customer types "Status of TXN-99213". Which engine will do better, and why?
Keyword wins easily. "TXN-99213" is a rare exact string — BM25 scores it very high and finds the one document containing it. The embedding model has probably never seen that ID, so it produces a vague vector and returns generic "transaction status" pages. Exact identifiers are keyword territory.
Concept 5 · M2.3 core
Hybrid search: run both, then merge
Two engines see different things. Hybrid search asks both, then combines the answers into one list.

Hybrid search is simple in idea. Send the query to the keyword index. Send the same query to the vector index. Now you have two ranked lists. Merge them into one.

The merge step is called fusion. That is the only new word here.

Concept 6 · Inside the merge
How fusion actually works
Two ways to merge two lists. You will use both in the lab.

Way 1 · Weighted score

Normalise both scores to 0–1. Then blend them with a weight.

final = α × semantic + (1 − α) × keyword

α = 0.5 means "trust both equally". Simple, tunable, but you must normalise carefully.

Way 2 · Reciprocal Rank Fusion (RRF)

Ignore scores completely. Use only the position in each list.

score = Σ 1 / (k + rank) k = 60

A doc ranked well by both engines wins. No normalisation needed. This is the enterprise default.

🧠 Quick check: why does RRF use 1/(60 + rank) instead of just 1/rank?
The 60 flattens the curve. With plain 1/rank, rank 1 scores 1.0 and rank 2 scores 0.5 — the top hit dominates everything, so a single over-confident engine wins alone. With 1/(60+rank), rank 1 scores 0.0164 and rank 2 scores 0.0161. Now agreement between the two engines matters more than one engine shouting loudly. That is exactly the behaviour you want from a merge.
🤯 Industry note: Elasticsearch, OpenSearch, Azure AI Search, Qdrant, Weaviate and Vespa all ship RRF as a built-in hybrid mode. You rarely write the formula yourself in production — but you must know what it does when the ranking looks wrong.

Concept 7 · M2.4 starts here
The retriever and what it returns
In LangChain you call one method. Behind it sits everything you just built.

A retriever is an object with one job: give it a question, it returns documents.

retriever = vectorstore.as_retriever(search_kwargs={"k": 20}) docs = retriever.invoke("My salary hasn't arrived") # docs → a list of 20 Document objects

Each Document has two parts: page_content (the text) and metadata (product, date, region, language).

🤔 Predict first: a production RAG bot answers using 5 documents. How many does the retriever usually fetch?
20 to 100. Retrieval is deliberately greedy. It casts a wide, cheap net so the right document is somewhere in the pile. A slower, smarter stage picks the final 5. Missing the document at stage one is unrecoverable — nothing downstream can invent it.
🤯 Two different goals: the retriever optimises for recall — "did we catch it at all?". The re-ranker optimises for precision — "is it at the top?". Different jobs, different tools, different speeds.

🧠 Quick check: why is a metadata filter often a bigger accuracy win than a better embedding model?
Because it removes whole classes of wrong answers before scoring even starts. If the customer is on a Retail account in India, no Corporate or US-region article can ever be returned — no matter how similar its wording is. A better model only reshuffles the same noisy pool. A filter deletes the noise. Filters are also nearly free and easy to explain to auditors, which matters in banking.
Concept 8 · The second opinion
Re-ranking: the first list is rarely the best
Retrieval was fast and rough. Now we spend real compute on 20 documents instead of 8,000.

Vector search compares two vectors that were made separately. The query never actually met the document.

A cross-encoder re-ranker is different. It reads the query and the document together, in one pass, and outputs one relevance score.

That is far more accurate. It is also far slower. So you only run it on the 20 survivors.

🧠 Quick check: what exactly does the LLM receive? Its own memory, or the documents?
Only the documents — pasted as plain text into the prompt, above the question. The LLM has no connection to your vector store, no memory of Meridian, and no way to look anything up. If the fact is not in those five documents, the model either says it does not know, or it invents something. Every retrieval bug becomes an answer bug.

⚠️ Five retrieval mistakes freshers make

  • Chunks too big — one chunk holds four unrelated topics
  • k too small — the answer was at position 12
  • No metadata filter — corporate answers to retail customers
  • Different embedding model for query and documents
  • No evaluation set — "it felt better" is not a metric

✅ Five practices that ship

  • Hybrid by default, semantic-only only if proven enough
  • Retrieve wide (k=20–50), re-rank narrow (top 3–5)
  • Filter on metadata before scoring
  • Log every retrieval — you cannot debug what you did not store
  • Measure Recall@20 and MRR on a fixed 50-question set
